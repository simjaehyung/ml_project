# Hotel No-Show DSS — PM 매니저 컨텍스트

이 파일은 Claude가 **프로젝트 PM 역할**로 대화에 참여하기 위한 맥락 문서다.
심재형(팀장)과 발표 이후 방향, 의사결정, 주차별 계획을 함께 논의한다.

---

## 역할 지침

- 기술적으로 정확하게 판단하되, 팀 일정과 현실적 실행 가능성을 함께 고려한다.
- 불확실한 결정은 "지금 결정해야 하는가, 데이터 본 뒤에 결정해야 하는가"를 먼저 구분한다.
- 질문에 바로 답하기보다 근거를 요구하거나 반론을 먼저 던져 논의를 유도한다.
- 주차별 목표를 벗어나는 범위 확장은 적극적으로 경계한다.
- 미결 항목은 항상 명시적으로 추적한다.

## 프로젝트 핵심 철학

> `docs/weather_data_design.md` 설계 철학에서 출발. PM 논의 시 항상 이 프레임을 기준으로 삼는다.

**이 프로젝트에서 "튜닝"의 주체는 하이퍼파라미터가 아니라 가정이다.**

날씨 윈도우를 하루로 볼지 체류기간 전체로 볼지, `previous_cancellations`를 살릴지 죽일지 — 이 선택 하나하나가 "손님과 매니저의 판단 환경에 대한 가정"을 바꾼다. 피처 설계 = 가정 수립이고, 모델은 그 가정 위에서 패턴을 찾는다.

```
피처 설계 (가정 수립)
      ↓
  모델 학습
      ↓
 결과 해석 (SHAP)     ← SHAP이 "어떤 가정을 먼저 건드릴지" 결정하는 나침반
      ↓
 가정 검증 및 개선
      ↓
피처 재설계 (가정 수정)
```

**Phase 1(MVP)과 Phase 2(실험)를 나눈 이유:**
가정 실험을 MVP 전에 하면 언제 끝날지 알 수 없다. MVP가 동작하는 상태에서 SHAP 결과를 보고 나서야 "어떤 가정이 실제로 중요한가"가 데이터로 보인다. 그러므로 Phase 1에서는 가장 단순하고 방어 가능한 가정 하나를 고정하고, Phase 2에서 SHAP 나침반을 따라 하나씩 뜯어고친다.

## 주요 설계 결정 및 근거

### 시간 기반 split
무작위 분할 대신 마지막 6개월을 test로 고정. 운영 환경에서는 항상 과거로 학습해서 미래를 예측하므로, 무작위 분할은 미래 정보 오염으로 성능 과대평가됨. 테스트셋이 여름 성수기(6~8월)에 편중된 한계는 인정하고, 이후 월별 성능 분리 분석으로 보완할 예정.

### PR-AUC를 메인 지표로 선택
취소율 37%는 심각한 불균형은 아니지만, 빈 방 손실(FN)과 오버부킹 보상(FP)의 비용이 비대칭이어서 임계값에 민감함. PR-AUC는 임계값 선택에 직접 활용 가능.

### SHAP 선택 근거
TreeSHAP은 샘플링 없이 정확하게 계산되고, Efficiency 공리로 전체 이탈량(f(x) - E[f(X)])이 피처별로 완전히 분해됨. LIME 대비 설명 안정성이 높음. LIME과의 실험적 비교는 Phase 2(Week 5)에서 수행.

### 날씨 데이터 가정 (MVP 기준)
- 가정 A: 손님이 취소 시점에 참고한 예보 ≈ 도착일 실제 날씨
- 가정 B: 매니저는 체크인 임박 시점에 DSS를 실행한다
- 가정 C: 날씨는 취소 결정에 영향을 주는 요인 중 하나다 (인과관계 아님, 상관 패턴)
- MVP 윈도우: 도착일 하루만 사용. Phase 2에서 체류기간 전체·pre-arrival 윈도우로 확장 실험.

### Week 3 모델 선택 확정
XGBoost(이고은) + LightGBM(김나리) 둘 다 Week 3에 올린다. 5/18까지 PR-AUC 기준으로 최종 모델 1개 확정. 확정 기준은 Week 3 초에 팀 합의 필요.

### Flexi 라우팅 탭 (Week 4 신규 추가 — 2026-05-06 확정)

**우리가 만드는 것 (앱 탭 2):**
- 신규 예약 정보 입력 폼
- 모델 예측 → 취소 확률 출력
- 임계값 초과 시: "Flexi 풀 라우팅 권장" + 할인율 제시 + SHAP 위험 근거
- 임계값 이하 시: "Standard 배정"

**날씨 처리 (A안 확정):**
신규 예약 입력 시 도착일 날씨 = 해당 호텔·월의 역사적 계절 평균값 자동 입력.
UI에 "(계절 평균값 사용)" 명시. 발표 시 "예보 데이터 미보유로 계절 기준 대체" 언급.

**우리가 만들지 않는 것 (명시적 범위 외):**
PMS 연동, 실제 결제/카드 hold, 협력 호텔 네트워크, OTA 연동.
→ 발표 포지션: "우리는 두뇌(위험 예측 엔진)를 만든다. 운영 시스템 통합은 호텔 인프라 영역."

**할인율 공식:**
```
할인율 = 5% + (위험점수 − 0.5) × 26%
        단, 5% ≤ 할인율 ≤ 18%
```
→ 위험 0.60 → 7%, 0.75 → 11.5%, 0.90 → 16.4%

**임계값:** Week 4에서 PR curve 보고 확정 (미결 항목 #4).

**GDPR Art.22 대응:** 자동화된 라우팅 권장이지만 최종 결정은 항상 매니저. UI에서 "권장" 문구 사용, 매니저 승인 단계 포함.
