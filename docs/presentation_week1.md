> 발표자: 심재형 (PM) | 기준일: 2026-05-03 | 형식: 문서 초안 (PPT 변환 전)

> **"우리는 취소 확률을 내는 게 아니라, 매니저가 지금 당장 무엇을 해야 하는지 알려주는 시스템을 만듭니다."**

## 슬라이드 구성

---

### SLIDE 1 — 표지

**제목**: 호텔 No-Show 의사결정 지원 시스템
**부제**: 예측 + 자동 해석 + 비즈니스 액션 권장
**팀**: 심재형 · 이고은 · 김나리

### SLIDE 2 — 문제 상황 (AS-IS)

**제목**: 호텔 매니저의 하루

**멘트**:
호텔 매니저는 매일 아침 이런 고민을 합니다.
"오늘 체크인 예정 손님 중 누가 안 올 것 같지? 오버부킹을 얼마나 걸어야 하지?"

현재는 이 판단을 **직관에만 의존**합니다.
데이터를 보면 예약 취소율이 평균 **37%** — 10명 중 4명이 안 옵니다.
City Hotel은 **41.7%**, Resort Hotel은 **27.8%**로 호텔 유형별로도 크게 다릅니다.

**슬라이드 내용**:
- 취소율 37% 강조
- "직관 의존 → 빈 방 손실 or 오버부킹 클레임" 구조도

---

### SLIDE 3 — 우리의 해답 (TO-BE)

**제목**: 단순 예측이 아닌 의사결정 지원

**멘트**:
많은 팀이 "취소 확률 0.73"을 출력하는 모델을 만듭니다.
근데 매니저 입장에서 0.73이 나오면 **뭘 해야 하는지** 모릅니다.

우리는 세 가지를 함께 출력합니다.

**슬라이드 내용 (코드 블록 형식)**:
```
입력: 예약 ID 12345
  - lead_time: 180일
  - previous_cancellations: 2회
  - deposit_type: No Deposit
  - 도착일 강수량: 25mm

출력:
  ① 취소 확률: 0.78 (HIGH)

  ② 주도 변수 Top 3 (SHAP 자동 해석)
     · 예약을 180일 전에 했음 (+0.21)
     · 과거 2회 취소 이력 (+0.15)
     · 선결제 없는 예약 (+0.12)

  ③ 권장 액션
     · 도착 30일 전 재확인 콜
     · 선결제 전환 유도 안내 발송
     · 오버부킹 후보로 분류
```

### SLIDE 4 — 데이터 소개

**제목**: 데이터 두 가지를 결합합니다

**멘트**:
기반 데이터는 두 가지입니다.

첫 번째는 Kaggle 호텔 예약 데이터.
포르투갈 City Hotel(리스본)과 Resort Hotel(알가르브) 2개 호텔,
2015년 7월부터 2017년 8월까지 **119,390건**의 예약 기록입니다.

두 번째는 Open-Meteo API로 수집한 **날씨 데이터**.
같은 기간, 같은 지역의 일별 기온·강수량·풍속 등 11개 변수입니다.
두 데이터를 호텔 + 도착일 기준으로 합쳤습니다.

**슬라이드 내용**:

| | 예약 데이터 | 날씨 데이터 |
|--|--|--|
| 출처 | Kaggle | Open-Meteo API |
| 기간 | 2015-07 ~ 2017-08 | 동일 |
| 규모 | 119,390행 | 793일 × 2개 호텔 |
| 주요 변수 | 32개 → 전처리 후 38개 | 기온, 강수량, 풍속 등 11개 |

### SLIDE 5 — 4가지 어려움 (개요)

**제목**: 시작 전에 이미 4가지 어려움을 발견했습니다

**멘트**:
저희 팀이 다른 팀과 다른 점 하나는,
**모델링 시작 전에 이미 이 데이터의 문제점 4가지를 먼저 발견하고 정의했다**는 겁니다.

이 4가지를 "이상함 → 조사 → 선택 → 한계" 구조로 정직하게 다루는 것이 이 프로젝트의 핵심입니다.

**슬라이드 내용**:
1. 데이터 누수 — 예약 후에 생기는 정보가 섞여 있다
2. 시간 비대칭성 — 학습 때 아는 것과 실전에서 아는 것이 다르다
3. 임계값 함정 — 0.5 기준이 비즈니스 현실과 맞지 않는다
4. 클래스 불균형 — accuracy로 평가하면 "무조건 정상" 전략에 진다

### SLIDE 6 — 어려움 ① 데이터 누수

**제목**: 어려움 ① — 예약 후에 생기는 정보

**멘트**:
원본 데이터 32개 컬럼 중 일부는 **예약 시점에는 존재하지 않는 정보**입니다.

예를 들어 `reservation_status`는 "체크아웃", "취소", "노쇼" 중 하나인데,
이건 **예약이 끝나고 나서야** 알 수 있는 값입니다.
이걸 그냥 넣고 학습하면 — 정확도 99%짜리 쓸모없는 모델이 나옵니다.

저희는 이 과정에서 **확정 drop 2개 + PM 추가 결정 drop 3개** 를 처리했고,
그 판단 근거를 하나하나 데이터로 검증해서 문서로 남겼습니다.

또 한 가지 흥미로운 사례: `days_in_waiting_list`.
대기 명단에 있다가 자리가 나서 확정된 일수인데 —
자리가 난 이유가 **다른 사람의 취소** 때문일 수 있습니다.
즉 이 변수에는 다른 사람의 미래 취소 정보가 간접적으로 녹아 있습니다.

**슬라이드 내용**:

| 처리 | 컬럼 | 이유 |
|------|------|------|
| 🔴 확정 drop | reservation_status / _date | 타깃과 동치 또는 미래 갱신값 |
| 🔴 추가 drop | assigned_room_type | 체크인 시점 배정값 |
| 🔴 추가 drop | days_in_waiting_list | 간접 미래 정보 포함 가능 |
| ⚠️ 조건부 keep | previous_cancellations | EDA 재검증 예정 |

---

### SLIDE 7 — 어려움 ② 시간 비대칭성

**제목**: 어려움 ② — 학습할 때 아는 것 vs 실전에서 아는 것

**멘트**:
날씨 데이터를 쓸 때 근본적인 문제가 있습니다.

학습 데이터엔 **도착일의 실제 날씨**가 있습니다.
하지만 실전에서 손님이 예약할 때는 도착일 날씨를 **모릅니다**.
평균 lead_time이 104일이니까, 104일 후 날씨를 예보로 알 수 있을까요? 불가능합니다.

저희가 선택한 방향:
- 일단 실제 날씨로 학습하되
- SHAP interaction 분석으로 lead_time이 짧을 때와 길 때 날씨의 영향이 달라지는지 정량화하고
- 이 한계를 발표에서 명시합니다

이게 저희 프로젝트의 **가장 흥미로운 분석 포인트**이기도 합니다.

**슬라이드 내용**:
```
학습 시점:  [예약 정보] + [도착일 실제 날씨 ✅] → 모델 학습
추론 시점:  [예약 정보] + [도착일 날씨 ??? ❌] → 예측 요청
                                    ↑
                           lead_time 평균 104일
                           → 일기예보로 알 수 없음
```
대응: 실제 날씨로 학습 + SHAP interaction(날씨 × lead_time) 분석 + 한계 명시

### SLIDE 8 — 어려움 ③④ (간략)

**제목**: 어려움 ③④ — 임계값과 평가지표

**멘트**:
세 번째는 임계값 문제입니다.
모델이 0.73을 출력했을 때 "취소"로 볼 것인가 "정상"으로 볼 것인가의 기준이 0.5인데,
이건 빈 방 손실과 오버부킹 클레임 비용이 같다는 가정입니다. 현실은 다릅니다.
저희는 PR curve 분석 후 0.35로 낮출 예정이고, 그 비즈니스 논리를 문서로 남깁니다.

네 번째는 평가지표입니다.
취소율이 37%인 데이터에서 accuracy로 평가하면
"모든 예약을 정상으로 예측"하는 전략이 63%를 기록합니다.
그래서 저희는 PR-AUC를 메인 지표로 쓰고, F1을 보조로 씁니다.

**슬라이드 내용**:
- ③ 임계값: 0.5 기본값 → PR curve 기반 0.35 조정 예정
- ④ 평가지표: accuracy 함정 → PR-AUC 메인, F1 보조

## 예상 Q&A

**Q1. 왜 날씨 데이터를 썼나요?**
> 취소 결정에 날씨가 영향을 준다는 가설이 있지만, 실제로 얼마나 중요한지는 데이터가 말해야 한다고 생각했습니다. SHAP 분석으로 날씨 변수의 실제 영향력을 정량화할 예정이고, 유의미하지 않다는 결과가 나와도 그것 자체가 의미있는 분석입니다.

**Q2. 포르투갈 데이터인데 일반화가 되나요?**
> 일반화보다는 이 두 호텔에서 어떤 변수가 중요한지를 정직하게 분석하는 것이 목표입니다. City/Resort 두 유형 비교 자체가 분석 포인트이기도 합니다.

**Q3. 이야기 생성과 액션 매핑은 어떻게 구현하나요?**
> SHAP 출력값을 기반으로 템플릿 방식으로 자연어를 생성합니다. 액션 매핑은 취소 확률 구간 + 주도 변수 조합에 따른 룰 테이블 방식입니다. 호텔 전문가 검증 없는 합리적 가설 수준임을 명시할 예정입니다.

**Q4. 5주 안에 가능한가요?**
> Week 4가 핵심이고 시스템 통합이 몰려 있습니다. PM이 Streamlit + 이야기·액션 모듈을 담당하고, Dev A·B가 각자 트랙을 병렬로 진행하는 구조라 가능하다고 판단했습니다.
