## 발표 스토리라인 (9~10분)

> 이 스토리를 기준으로 모든 개발과 발표 준비가 정렬된다.

```
[1분] 문제 — 호텔이 왜 손해를 보는가
  → City Hotel 취소율 41.7%, Resort Hotel 27.8%
  → 취소 = 빈 방(수익 손실) + 오버부킹(고객 불만) + 음식 낭비(운영 손실)
  → "매니저는 오늘 어떤 예약이 위험한지 모른다"

[1분] 우리의 접근 — 데이터로 보는 취소의 구조
  → 119,390건 예약 + 날씨 데이터 (가정 A·B·C)
  → 시간 기반 분할 (왜 무작위 분할 안 쓰나)
  → 핵심 결정: deposit_type 제거, country Top10+Other
  ★ 김나리 발표 파트

[2분] 모델 — 무엇이 취소를 예측하는가
  → Dummy → LR → RF → XGBoost → LightGBM PR-AUC 비교 차트
  → 최종: LightGBM 0.8189 (Dummy 대비 +112%)
  → SHAP top 피처: 포르투갈 국적, 특별 요청 없음, 120일+ 선예약
  → "모델이 아닌 가정을 튜닝한다"는 철학
  ★ 김나리 발표 파트

[3분] 플랫폼 데모 — 어떻게 활용하는가 (라이브)
  탭1: "오늘 고위험 예약 87건. 이 예약 클릭 — 왜 위험한가?"
       → SHAP waterfall: 포르투갈 국적 +0.31, 120일 선예약 +0.18
  탭2: "이 신규 예약, 취소 확률 81%. Flexi 권장: 13.1% 할인"
       → 할인 적용 ADR, 총 절감액, 주요 위험 요인
  ★ 이고은 데모 파트

[2분] 검증 — 이 할인율이 실제로 작동하는가
  → LLM 에이전트 ~16,000건 시뮬레이션 (AgentSociety 기법)
  → 임계값 0.60에서: 수락률 X%, RevPAR +Y%, walk_rate Z%
  → "우리의 할인율 공식이 실제 고객 반응에서도 유효하다"
  ★ 심재형 발표 파트

[1분] 마무리 + Phase 2 예고
  → 증명한 것: 이 데이터셋에서 취소 위험 우선순위화 유효
  → Phase 2: 날씨 윈도우·피처 가정 실험 예정
```

## 라이브 데모 시나리오 (이고은 준비)

**탭1 시연 (90초):**
1. 앱 열기 → 탭1 표시 (KPI: 고위험 87건, 손실 €12,430)
2. 임계값 슬라이더를 0.60 → 0.75로 조정 → 건수 변화 확인
3. 테이블에서 취소확률 0.91 예약 클릭
4. 우측 SHAP waterfall: "포르투갈 국적, 120일 선예약이 주요 요인"

**탭2 시연 (90초):**
1. 탭2 클릭 → 입력 폼 표시
2. 국적: PRT, 채널: Online TA, 선예약: 120일, ADR: €95, 특별요청: 0 입력
3. [취소 위험 분석] 클릭
4. 결과: 취소 확률 81%, "Flexi 풀 라우팅 권장 — 할인율 13.1%, 적용 ADR €82"
5. "최종 결정은 매니저 확인 필요 (GDPR Art.22)" 문구 언급

## 김나리 Q&A 준비 — 6개 핵심 질문

| # | 예상 질문 | 답변 핵심 |
|---|-----------|-----------|
| 1 | 왜 PR-AUC인가? | 빈 방 손실 vs 오버부킹 비용이 비대칭 → 임계값 선택이 중요 → ROC-AUC 부적합 |
| 2 | 왜 LightGBM 선택? | PR-AUC 0.8189 vs XGBoost 0.8053, 차이 0.0136 ≥ 0.01 기준 초과 |
| 3 | country_PRT가 1위인 이유? | 포르투갈인의 자국 호텔 취소 패턴 — SHAP beeswarm에서 방향성 확인 |
| 4 | deposit_type 왜 제거? | Non Refund 예약 99.2% 취소 → 예약 시점에 없는 정보, 사후 기록 오염 |
| 5 | 날씨 데이터 왜 추가? | 가정 C: 날씨가 취소 결정에 영향 — SHAP에서 temperature_2m_min이 top 15에 진입 |
| 6 | 시간 기반 분할 이유? | 운영 환경에서는 항상 과거로 학습·미래 예측 → 무작위 분할은 미래 정보 오염 |
