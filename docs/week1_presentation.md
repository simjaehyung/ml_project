## Slide 02 — THE PROBLEM

**헤드라인:** 빈 방과 오버부킹, 둘 다 매출 손실이다

**수치:**
- City Hotel (리스본) 취소율 **41.7%**
- Resort Hotel (알가르브) 취소율 **27.8%**
- 예약 10개 중 약 4개가 사라진다

**핵심 구조:**

| 상황 | 행동 | 결과 |
|------|------|------|
| 취소 예약을 놓침 | 개입 안 함 | 빈 방 → 매출 손실 |
| 정상 예약에 개입 | 잘못 건드림 | 오버부킹 → 보상 비용 |
| **위험 예약을 식별** | **손님과 합의 → Flexi** | **양쪽 모두 이득** ← NEW |

> 어느 예약이 위험한지 미리 알 수 없다면 둘 중 하나는 항상 틀린다.

**⚑ 용어 정의 (발표자 필수 숙지 — 교수님 지적 항목)**

| 용어 | 정의 | 예측 가능 여부 |
|------|------|--------------|
| **취소 (Cancellation)** | 체크인 전 손님이 사전 통보하고 예약을 철회 | ✅ 사전 예측 가능 — **우리가 예측하는 것** |
| **No-show** | 예약 후 아무 연락 없이 체크인 당일 나타나지 않음 | ❌ 사후에만 확인 가능 — 예측 대상 아님 |

> 우리의 타깃 변수는 `is_canceled` (취소 여부)다.
> "Hotel No-Show DSS"라는 제목은 취소·no-show 모두를 포함하는 빈 방 문제 전체를 지칭하며,
> 실제 예측 대상은 사전에 개입 가능한 **취소**로 한정한다.

*질문 받을 경우 → "No-show는 당일 사후 확인이라 사전 개입 여지가 없고 데이터셋의 타깃도 is_canceled입니다. 예측 가능한 취소에 집중했습니다."*

## Slide 03 — AS-IS

**헤드라인:** 지금 판단의 주체는 데이터가 아니라 경력과 감이다

**"이 예약, 받을까 말까."**

매니저는 하루에도 수십 번 이 질문을 스스로에게 던진다.

- **기준 없음:** 어떤 예약이 위험한지 판단하는 기준이 없다
- **기록 없음:** 왜 그렇게 판단했는지 남는 것이 없다
- **설명 불가:** 같은 매니저도 날마다 다른 결론이 나온다

> **오늘 먼저 확인해야 할 예약이 어떤 건지 어떻게 아는가?"**

**기존 접근과의 차이:**

| 기존 방식 | 우리 방식 |
|----------|----------|

| 매니저가 골라서 확인 | 시스템이 먼저 걸러서 제시 |
| 숫자만 출력 | 확률 + 이유 + 권장 행동 |

**헤드라인:** 확률 · 근거 · 행동을 묶어서, 우선순위 리스트로

## Slide 06 — 데이터 범위 & 프로젝트 목표

**헤드라인:** 우리가 증명하려는 것은 이 접근법이 실제로 작동한다는 것이다

**데이터:**

| 항목 | 내용 |
|------|------|
| 출처 | Kaggle Hotel Booking Demand |
| 대상 | 포르투갈 2개 호텔 (리스본 City Hotel / 알가르브 Resort Hotel) |
| 기간 | 2015년 7월 ~ 2017년 8월 (26개월) |
| 규모 | 119,390건 |
| 외부 데이터 | Open-Meteo API — 도착일 기준 날씨 결합 |

**작동의 기준:**
1. 모델이 baseline 대비 유의미하게 취소를 예측한다 (PR-AUC)
2. SHAP이 그 이유를 설명 가능한 언어로 분해한다
3. 매니저가 실제로 행동할 수 있는 출력이 나온다

> 다른 호텔·지역으로의 일반화는 이 프로젝트의 목표가 아니다.

## Slide 07 — OUTPUT 정의

**헤드라인:** 시스템이 주는 것과 매니저가 하는 것을 분리한다

```
INPUT   예약 정보 + 도착일 날씨
           ↓
        [모델]
           ↓
OUTPUT  PROBABILITY   취소 확률 0.84
        EXPLANATION   lead_time 길다      +0.21
        (SHAP)        이전 취소 이력 있다  +0.15
                      보증금 없음          +0.09
                      ────────────────────────
                      합산 +0.47  (기준값 0.37 → 예측 0.84)
        ACTION        → 사전 연락 권장
           ↓
매니저  연락 보낼지 / 오버부킹 조정할지 / 넘길지 — 최종 결정은 매니저 권한
```

**SHAP을 쓰는 이유:**
TreeSHAP은 샘플링 없이 정확하게 계산. 전체 예측 이탈량(f(x) - E[f(X)])이 피처별로 완전히 분해되어 설명이 누락 없이 합산된다.

## Slide 08 — TIME SPLIT

**헤드라인:** 미래를 학습에 쓰지 않는다

- 전체 기간: 2015년 7월 ~ 2017년 8월 (26개월)
- Train: 2015년 7월 ~ 2017년 2월 (앞 20개월)
- Test: 2017년 3월 ~ 2017년 8월 (마지막 6개월) ← 고정

**무작위 분할이 아닌 시간 기반 분할:**
운영 환경에서는 항상 과거로 학습해서 미래를 예측한다. 무작위 분할은 미래 정보가 학습에 섞여 성능을 과대평가한다.

## Slide 09 — 전처리 — 4가지 어려움

**헤드라인:** 모델링 전에 이미 발견한 4가지 어려움

| # | 어려움 | 내용 |
|---|--------|------|
| 1 | 데이터 누수 | 일부 컬럼이 체크인·체크아웃 후에 채워지는 값 — 정답을 미리 보여주는 것 |
| 2 | 시간 비대칭 | 학습 시 실제 날씨 사용 / 예약 시점엔 예보뿐 — 입력의 성격이 다름 |
| 3 | 임계값 함정 | 빈 방 손실과 오버부킹 비용이 비대칭 — 0.5 기준이 최적이 아님 |
| 4 | 클래스 불균형 | 취소(37%) < 정상(63%) — 정확도 지표가 왜곡됨 |

**지표 선택:** PR-AUC (메인) / F1@0.5 (보조)

## Slide 10 — 누수 컬럼 제거

**헤드라인:** 예약 후 채워지는 값은 모두 제거한다

**확정 누수 (2개):**

| 컬럼 | 이유 |
|------|------|
| `reservation_status` | 타깃(취소 여부)과 100% 동치 — 정답을 그대로 넣는 것 |
| `reservation_status_date` | 취소/체크아웃 처리 시점 기록 — 예약 시엔 존재하지 않는 미래 정보 |

**시점 어긋난 컬럼 (2개):**

| 컬럼 | 이유 |
|------|------|
| `assigned_room_type` | 체크인 시 배정 — 예약 시점에 모르는 값 |
| `days_in_waiting_list` | 타인의 취소가 결정 — 타인 미래 정보 간접 반영 |

**추가 제거 (1개):**
`previous_bookings_not_canceled` — 재방문자 25.5%가 값 0, 모순 + 신호 없음

## Slide 12 — 날씨 포함 근거 & 가정

**헤드라인:** 손님은 도착일 날씨를 보고 취소를 결정한다

**포함 근거:**
손님이 취소를 결정하는 시점은 예약 당일부터 체크인 직전까지 어디든 가능하다. 그 판단에는 자연스럽게 "도착했을 때 날씨가 어떨까"가 포함된다. 모델은 **어떤 날씨 조건의 도착일을 가진 예약이 취소됐는가**를 학습한다.

**명시적 가정:**

| 가정 | 내용 | 한계 |
|------|------|------|
| A | 손님이 본 예보 ≈ 실제 날씨 | 예보 데이터 미보유 — 실제 날씨로 대체 |
| B | 매니저는 체크인 임박 시점에 시스템을 실행 | 조기 실행 시 날씨 신뢰도 낮아질 수 있음 |
| C | 날씨는 취소 결정에 영향을 주는 요인 중 하나 | 인과관계 아님 — 상관 패턴으로 학습 |

**현재 윈도우:** 도착일 하루만 사용. SHAP 결과 보고 확장 여부 결정.

## Slide 13 — 시스템 흐름 & 마무리

**헤드라인:** 예약 정보가 매니저의 행동으로 이어지는 구조

```
예약 정보 입력
+ 도착일 날씨
      ↓
   모델 예측
  (취소 확률)
      ↓
  SHAP 분석
 (기여 요인 분해)
      ↓
우선순위 리스트 출력
      ↓
매니저 → 연락 / 오버부킹 조정 / 보증 정책 검토
신규 예약 → 위험도 기반 Flexi 풀 라우팅 (확장 시나리오)
```

**이 프로젝트가 증명하는 것:**
1. 취소 위험 예약을 데이터로 먼저 찾을 수 있다
2. 그 이유를 매니저가 읽을 수 있는 언어로 설명할 수 있다
3. 확률에서 행동으로 이어지는 구조를 만들 수 있다

> 이 접근법이 해당 데이터셋에서 유효하게 작동함을 이 프로젝트로 증명한다.

## Slide 14 — FLEXI ROUTING (확장 시나리오)

**헤드라인:** 투명한 위험 기반 예약 — Flexi Rate

> 손님이 알고 동의한 위험. 호텔과 손님이 정보를 공유한다.

**라우팅 구조:**

```
신규 예약 요청
      ↓
모델 위험 예측
      ↓
┌─────────────────────────┐
↓                         ↓
Low / Med 위험             High 위험 (≥ 임계값)
Standard                  Flexi Rate
정가 · 즉시 보장           −8~18% · D-1 확정
```

**할인율 공식:**
```
할인율 = 5% + (위험점수 − 0.5) × 26%   (5% ≤ 할인율 ≤ 18%)
예: 위험 0.75 → 할인 11.5%
```

**손님 보호:**
- D-1 18:00 확정 통보
- 미배정 시: 동급 호텔 + 1박 환불 + €30 크레딧

**명시적 동의:**
- 예약 흐름의 별도 단계 (약관 체크박스 X → 안내 후 동의)
- GDPR Art.22 준수 — 최종 결정은 항상 매니저 권한

**우리가 만드는 것 / 만들지 않는 것:**

| 항목 | 범위 |
|------|------|
| 위험 예측 엔진 | ✓ 이 프로젝트 |
| Flexi 라우팅 권장 UI | ✓ 이 프로젝트 |
| SHAP 위험 근거 출력 | ✓ 이 프로젝트 |
| PMS 연동 · 결제 시스템 | 호텔 인프라 |
| 협력 호텔 네트워크 | 호텔 인프라 |
| OTA 연동 | 단계적 확장 |

> **우리는 두뇌를 만든다.**
> 운영 시스템 통합은 호텔의 PMS · 결제 인프라 영역이다.
> 이 선이 "아이디어"와 "운영 가능한 시스템"을 가르는 경계다.

### Q1. "문제 정의가 모호하다. 얘들이 뭘 하겠다는 건지 모르겠다."

**A.**
"저희가 정의하는 문제는 두 층위입니다.

첫째, 기존 예약 관리입니다. 체크인이 다가오는 예약들 중 오늘 매니저가 먼저 확인해야 할 예약이 어떤 건지를 데이터로 찾아 위험 순위 리스트로 제시합니다. 단건 확률 조회가 아니라 전체 예약을 정렬해서 드립니다.

둘째, 신규 예약 라우팅입니다. 예약 요청이 들어오는 시점에 같은 위험 예측 엔진을 적용해서, 고위험 예약은 손님에게 사전 안내하고 동의를 받은 뒤 Flexi Rate 풀로 분기합니다. 손님은 할인을 받는 대신 D-1에 객실 확정 통보를 받고, 미배정 시 보상 패키지를 받습니다.

즉 하나의 위험 예측 엔진이 두 가지 문제를 동시에 해결합니다. 기존 예약에서는 우선순위를 잡아주고, 신규 예약에서는 위험도에 따라 상품 자체를 분기하는 것입니다."

→ **Slide 04** (문제 정의) → **Slide 14** (Flexi 구조) 순으로 가리키며 답변

### Q2. "바운더리가 어디냐. 어느 호텔, 어느 지역, 어느 데이터냐."

**A.**

### Q5. "No-show랑 취소가 다른 거 아니냐. No-show는 예측할 필요가 없는 거 아니냐."
